# architecture.md

## 技術スタック

- デスクトップ基盤: Tauri v2
- バックエンド: Rust
- フロントエンド: React
- 言語: TypeScript
- 対象 OS: Windows / macOS

## 責務分離

### Rust / Tauri

- タイマー主制御
- セッション進行管理
- 状態遷移管理
- 設定の永続化
- 通知、トレイ、ウィンドウ制御などの OS 連携

### React

- 状態表示
- ユーザー入力受付
- バックエンド状態の購読と描画
- `.design/design.pen` に忠実な UI 構築

## タイマー設計方針

- 時間経過の真実は Rust 側に置く
- React 側は表示更新に必要な最小限の情報だけ受け取る
- タイマーは `start / pause / resume / reset / skip` を明示的なイベントとして扱う
- Focus、Short Break、Long Break の状態遷移を状態機械として整理する
- 復帰時やスリープ後も破綻しないよう、経過時刻ベースで補正可能な設計を採る

## 状態モデル

想定する主要状態は以下です。

- 現在のセッション種別
- セッション残り時間
- 実行状態
- 完了済み Focus 回数
- Long Break 発火条件
- ユーザー設定

React 側に複製 state を増やさず、必要な派生表示は描画時に計算することを基本とします。

## 永続化方針

- 永続化対象は設定と、必要最小限のセッション情報に限定する
- 一時的な UI 状態や派生値は永続化しない
- フォーマットは単純で移行しやすいものを選ぶ
- 保存失敗時のフォールバックとデフォルト値を明確にする

## OS 差分の扱い

- 通知、トレイ、ウィンドウ制御は OS ごとに差分が出る前提で設計する
- 差分吸収は Rust / Tauri 側で行い、React 側に OS 分岐を持ち込みすぎない
- Windows / macOS の慣習差を踏まえつつ、UI の基本構造は統一する

## パフォーマンス方針

- idle 時 CPU 使用率を最小に保つ
- 常駐時メモリを抑える
- 起動時に不要な初期化を行わない
- フロントエンドに高頻度計算や高頻度 state 更新を持ち込まない

## 非目標

- タスク管理や分析ダッシュボードの肥大化
- クラウド同期のような重い拡張
- 過度なテーマ機能や演出
- Web アプリ前提の実装をそのまま持ち込むこと
