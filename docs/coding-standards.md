# coding-standards.md

## TypeScript / React 実装規約

- TypeScript は strict mode を前提とする
- `any`、曖昧な union、過剰な型アサーションを避ける
- React は表示層中心に使い、タイマー主制御は持たせない
- state は最小限にし、派生 state を増やさない
- コンポーネントは画面構造に沿って分割し、責務の境界が読めるようにする
- 重い UI ライブラリや不要な global state ライブラリは原則導入しない

## Rust / Tauri 実装規約

- タイマー、状態遷移、永続化、OS 連携は Rust 側で責務を分離する
- 状態遷移は enum と明示的なイベントで表現する
- UI の都合だけの分岐をコアロジックに混ぜない
- エラーは握りつぶさず、呼び出し側が判断できる形で返す
- `cargo fmt` と `cargo clippy` を通すことを前提にする

## UI 実装規約

- `.design/design.pen` を最優先の正本として扱う
- タイマー表示を主役にし、情報密度をむやみに増やさない
- Focus は青系、Break は緑系の状態差を守る
- 余白、整列、視線誘導、ボタンの主従を崩さない
- hover / active / disabled の状態は実装するが、過剰に装飾しない
- アニメーションは最小限にする

## 依存追加ルール

- 依存追加は明確な必要性がある場合だけ行う
- 追加前に代替案、サイズ影響、起動時間影響、保守性影響を整理する
- 1 回の変更で依存追加と無関係なリファクタを同時に行わない
- フロントエンド依存は特に慎重に扱う

## コメント方針

- コードを読めば分かる説明コメントは書かない
- 状態遷移、OS 差分、タイマー補正など読み取りにくい意図だけを短く補足する
- ドキュメント化すべき設計判断は `docs/` または `.agents/templates/` に残す

## リファクタ方針

- 振る舞いを変えない変更だけをリファクタとして扱う
- 機能追加とリファクタは原則分ける
- 無関係な命名変更やファイル移動を混ぜない
- パフォーマンス改善は計測または観察結果とセットで行う

## 変更前の明示事項

大きな変更前には最低限以下を整理すること。

- 何を変えるか
- なぜ変えるか
- `.design/design.pen` との関係
- パフォーマンスへの影響
- 依存追加の有無
